# Issues #136 and #137 — drop default values from Device and Component constructors

This document tracks the campaign for two related defects in
`MTConnect.NET-Common`:

- **#136** — `new Device()` auto-generated a fresh random `Uuid` per
  construction. Consumers that did not explicitly set `Uuid` silently
  violated the MTConnect XSD `uuid` identity contract: each
  reconstruction produced a new identity, which in production caused one
  historian or dashboard row per connector restart against the same
  `Device.Name`.
- **#137** — adjacent constructor-default defects in two POCO families:
  - **A.** `Device` default constructor stamped placeholder values
    (`Id` from `StringFunctions.RandomString(10)`; `Name = "dev"`).
  - **B.** Every `*Component` subclass (115 generated `*.g.cs` files
    plus the SysML-import Scriban template) back-filled `Name` from the
    lowercase type name (e.g. `Name = "axes"`, `Name = "linear"`).

The two issues land as a single PR per
`extra-files.user/plans/landing-coordination.md` because both modify the
same `Device` default constructor; splitting them would guarantee a
rebase conflict between the two PRs.
